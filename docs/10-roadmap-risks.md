# 10 — Roadmap, Risks & Future Enhancements

Phases match the agreed implementation workflow. Each phase ends with: architecture review, refactor pass, test-coverage check, documentation update. Estimates assume 2–3 engineers.

| **2. ## 2. Risk analysis

| Risk                                                                                           | L   | I   | Mitigation                                                                                                                                                                                 |
| ---------------------------------------------------------------------------------------------- | --- | --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Snapshot data growth outpaces plan** (millions of pages × daily)                             | H   | H   | Designed-in: partitions + retention tiers + carry-forward rows for unchanged pages (a 1M-page site with 2% churn writes ~20k full snapshots/day, not 1M); monitor bytes/crawl from Phase 3 |
| **Render capacity becomes the bottleneck** on JS-heavy properties                              | M   | H   | HTTP-first with auto-detect; per-site render budgets; render pool scales independently; worst case: schedule render-heavy sites off-peak                                                   |
| **Google rich-result requirements drift** → stale verdicts                                     | H   | M   | Requirements are versioned data packs; quarterly pack-refresh procedure with golden-test diff; verdicts display pack version                                                               |
| **Anti-bot/WAF on own properties blocks the crawler**                                          | M   | M   | Dedicated UA + published crawler IPs for infra allowlisting; circuit breaker prevents alert storms; escalation runbook                                                                     |
| **Duplicate/simhash false positives erode trust**                                              | M   | M   | Conservative default threshold, per-project tuning, "confidence" surfaced, easy ignore workflow                                                                                            |
| **Issue fingerprint instability** (URL churn, evidence-key changes) breaks workflow continuity | M   | M   | Fingerprint spec frozen in Phase 4 with migration story; fingerprints derived from normalized stable keys, versioned like scoring                                                          |
| **One-team scope creep** (830k checks/day tempts feature requests: rankings, CWV…)             | H   | M   | Non-goals stated in docs; new scope = new phase, not silent additions                                                                                                                      |
| **AI costs/latency** on large crawls                                                           | L   | L   | Explanations on-demand + cache-by-evidence; prioritization is top-K only                                                                                                                   |
| **Postgres write saturation at fleet scale**                                                   | L   | H   | Batch inserts, partition-local indexes; headroom plan: table-per-month already shards time; if needed, per-project DBs are possible because nothing joins across projects                  |
| **Key-person dependency**                                                                      | M   | M   | ADRs, runbooks, this doc set kept current as a phase-exit criterion                                                                                                                        |

## 3. Future enhancements (explicitly out of v1)

- **Search Console integration** — join issues with impressions/clicks to weight priority by traffic.
- **Log-file analysis** — Googlebot crawl-budget insights from CDN/server logs.
- **JavaScript-diff rendering audit** — compare static vs rendered DOM to quantify JS-dependence of SEO tags.
- **Public status API + CI gate** — block deployments if a pre-prod crawl introduces critical regressions ("SEO tests" in the release pipeline).
- **Jira/Linear sync** — issue-state → ticket lifecycle.
- **Sitemap-priority-weighted scoring** and traffic-weighted site scores.
- **Multi-region crawling** for geo-targeted content and hreflang serving verification.
- **CLI** (`seo-guardian crawl --url …`) reusing the pure engines for local pre-commit checks.
- **SSO (SAML/OIDC)** for enterprise auth; SCIM provisioning.
- **Anomaly detection** on trends (alert on statistical outliers, not just thresholds).
